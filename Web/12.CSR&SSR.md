> SSR : Server side Rendering

### SPA(Single Page Application)

> 최초에 페이지 전체를 로딩한 뒤, 데이터만 변경하여 사용할 수 있는 애플리케이션. SPA는 기본적으로 페이지 로드가 없고, 모든 페이지가 단순히 HTML5 History에 의해 렌더링 됨.
> 

기존의 전통적 방식인 SSR에는 성능 문제가 있음

→ 요청마다 새로고침이 발생하면서 페이지를 로딩할 때마다 서버로부터 리소스를 전달받아 해석하고 화면에 렌더링하는 방식인 SSR은 웹에서 제공하는 데이터가 많아질 수록 성능문제가 발생

→ **인터랙션이 많이 환경에서 비효율적**. 렌더링을 서버쪽에서 진행하면 서버 자원이 그만큼 많이 사용되기 때문에 불필요하게 **트래픽이 낭비**됨

**CSR 방식**은 사용자의 행동에 따라 필요한 부분만 다시 읽어옴. 즉 **SSR보다 빠른 인터랙션**을 기대할 수 있다. **서버는 단지 JSON 파일만 보내주고**, HTML을 그리는 역할은 자바스크립트를 통해 클라이언트 측에서 수행, 즉 **클라이언트 측에서 렌더링**된다.

뷰 렌더링을 클라이언트의 브라우저가 담당하고, 웹앱을 브라우저에게 로드한 다음 데이터만 전달받아 보여주는 CSR은 트래픽을 감소시키고, 사용자에게 더 자은 경험을 제공

⇒ 최근에는 SSR과 CSR 방식을 모두 사용한 하이브리드 렌더링을 많이 사용함.

> 장단점 정리
> 

**CSR**

- 장점
    - 트래픽 감소 : 필요한 데이터만 받음
    - 사용자 경험 : 새로고침이 발생하지 않음. 사용자가 네이티브 앱과 같은 경험을 할 수 있음
    
    (네이티브 앱이란? 모바일에 최적화된 앱)
    
- 단점
    - 검색 엔진 : 검색엔진의 크롤러가 데이터 수집를 수집하는데 어려움이 있음(초기 HTML에는 최소한에 정보만 포함되어 있고, 나머지 콘텐츠는 JS가 실행된 후 브라우저에서 동적으로 로드되기 때문)
    
    (크롤러란? 검색엔진이 웹사이트의 정보를 자동으로 수집하기 위해 사용하는 프로그램)
    

**SSR**

- 장점
    - 검색엔진 최적화
    - 초기로딩 성능개선 : 첫 렌더링된 HTML을 클라이언트에서 전달해주기 때문에 초기로딩속도를 많이 줄여줌
- 단점
    - 프로젝트 복잡도 : 라우터 사용하다보면 복잡도 높아질 수 있음
    - 성능 악화 가능성
